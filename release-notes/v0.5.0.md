# Gigabrain v0.5.0

## Summary

`v0.5.0` is the Memory OS release.

Gigabrain now combines hybrid native memory capture, a world-model projection layer, strategy-aware recall orchestration, an Obsidian-first curated surface, and stronger nightly operations into one coherent plugin release.

## Highlights

### World model and orchestrated recall

`v0.5.0` adds a world-model layer on top of atomic memories:

- entities
- aliases
- beliefs
- episodes
- open loops
- contradictions
- syntheses

Recall no longer treats every query as a flat search. The orchestrator now chooses between quick context, entity brief, relationship brief, timeline brief, and verification-oriented lookup so the plugin can answer people-, time-, and continuity-heavy questions more deliberately.

### Hybrid memory architecture

The `v0.5` architecture is intentionally local-first and operationally simple:

- native markdown stays the human-readable source layer
- SQLite stays the structured registry and query layer
- FTS5 accelerates lexical recall inside SQLite
- there is no separate vector database requirement for core capture and recall

Durable remembers can land in both native markdown and the structured registry. Ephemeral remembers can stay daily-note-only by default. Native notes are indexed into SQLite for unified recall and can be promoted back into structured memories when they look durable enough.

### Curated Obsidian surface

The Obsidian surface is now a real operator-facing memory product instead of a raw dump:

- `00 Home` for the landing page
- `10 Native` for mirrored native memory
- `20 Entities` and `20 Nodes/active`
- `30 Views` for current state, what changed, people, projects, contradictions, and more
- `40 Reviews`, `50 Briefings`, and `60 Reports`

The runtime workspace remains the source of truth. The Obsidian surface is a generated, read-only browse layer with protected human folders such as `Inbox/` and `Manual/`.

## Operational improvements

`v0.5.0` also hardens the operating path:

- nightly maintenance rebuilds FTS5 after `VACUUM`
- nightly now runs `graph_build` after `vault_build`
- month-specific temporal recall prefers source-dated memories instead of generic rows whose `updated_at` only happens to fall in the same month
- entity and world-model projection suppress common metadata noise tokens
- Gigabrain now declares itself as a `memory` plugin so OpenClaw can register it cleanly in the memory slot
- fresh-workspace nightly runs no longer fail if `memory_relations` has not been created yet

## What changed for users

- Obsidian is recommended but not required.
- The plugin can be used without a local embedding setup for core write/recall behavior.
- The setup wizard enables the `v0.5` defaults and builds the first surface automatically.
- Manual OpenClaw setups should point `plugins.slots.memory` at `gigabrain`.

## Validation

Validated for this release candidate on **March 11, 2026**:

- full local test suite passing (`npm test`)
- package audit via `npm pack --dry-run`
- live Nimbus recall/write/nightly/vault checks
- fresh-workspace setup, native write, native sync, orchestrated recall, nightly, and vault doctor smoke checks

## Known release note

Gigabrain `v0.5.0` is release-ready at the plugin level, but npm audit still reports upstream high-severity findings through the current `openclaw@2026.3.8` dependency chain. At the time of writing, that is an upstream dependency situation rather than a Gigabrain runtime regression.
